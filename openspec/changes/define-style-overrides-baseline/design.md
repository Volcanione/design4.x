# Design: define-style-overrides-baseline

## 上下文

当前组件库已经具备 `YoButton`、`YoSpace`、`YoConfigProvider`、按需引入、resolver 和预览站。

接下来需要持续补齐更多 AntDV 组件封装，同时还要支持 Yostar 自定义 props 和业务样式。如果样式覆盖规则不先明确，组件越多，返工越大。

## 设计决策

采用“四层样式覆盖模型”：

1. Ant Design Vue 原始样式负责基础结构、状态和交互。
2. `YoConfigProvider` token 负责全局视觉基线。
3. Yo 组件稳定类名负责业务语义样式。
4. AntDV 内部选择器只作为最后手段，并且必须限制在 Yo 根类名下。

采用“样板组件先行”的迭代方式：

- 先以 `YoButton` 和 `YoSpace` 固化基础模式。
- 后续每新增一个组件，同步完成实现、样式、导出、resolver、文档和 demo。
- 每完成一个组件类别后，再做一次统一视觉收敛。

## 备选方案

### 先补完组件，再统一改样式

未采用。该方式前期推进快，但后续会集中暴露 class 命名、props 分层、文档和样式产物不一致的问题，返工成本高。

### 先完整设计全部样式，再补组件

未采用。复杂组件的真实问题只有实现后才会暴露，提前一次性设计全部样式容易过度设计。

### 大量全局覆盖 `.ant-*`

未采用。组件库需要支持与原生 AntDV 混用，全局覆盖会影响使用方原本的 `a-*` 组件。

## 风险

- 如果 Yo 根类名遗漏，样式无法稳定收敛。
- 如果 fake props 没有从 attrs 中剥离，可能污染 DOM 或触发 Vue warning。
- 如果过早深度覆盖 AntDV 内部 DOM，后续 AntDV 版本升级风险会变高。
- 如果组件文档没有同步属性表，使用方无法判断哪些 props 是 AntDV 原生、哪些是 Yo 扩展。
